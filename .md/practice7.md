* * Відповіді на контрольні запитання (для звіту)
1) Поясніть принципову різницю між URL-параметрами шляху (Path Parameters) та параметрами запиту (Query Parameters). В яких випадках який підхід слід використовувати?

* Path Parameters (наприклад, /users/42) використовуються для ідентифікації конкретного ресурсу або об'єкта. Їх слід використовувати, коли параметр є обов'язковою частиною маршруту (наприклад, щоб відкрити сторінку конкретного поста або профілю).

* Query Parameters (наприклад, /users?sort=asc) використовуються для фільтрації, сортування або пагінації існуючого списку ресурсів. Вони є необов'язковими і не змінюють сам ресурс, а лише модифікують його відображення.

2) Що робить хук useParams() і який обов'язковий синтаксис необхідно використати в визначенні <Route path="..."> для того, щоб цей хук запрацював?
Хук useParams() витягує динамічні параметри шляху з поточного URL у вигляді об'єкта. Щоб він запрацював, у визначенні маршруту необхідно використати синтаксис з двокрапкою перед назвою змінної, наприклад: <Route path="/post/:postId" element={<PostDetails />} />. Тоді useParams().postId поверне значення з URL.

3) Чому зберігання стану пошукового рядка (Search Bar) в URL (через useSearchParams) вважається архітектурно кращим рішенням, ніж зберігання його у локальному стані компонента (useState)? Опишіть переваги для кінцевого користувача.
Це критично важлива бізнес-вимога, оскільки URL виступає як "Єдине джерело істини" (Single Source of Truth). Основні переваги для користувача:

* Shareability: Користувач може скопіювати посилання з результатами фільтрації та надіслати його в месенджері іншому користувачу.

* Bookmarks: Можна зберегти специфічний пошук у закладки браузера.

* History: Дозволяє коректно працювати кнопкам браузера "Назад" та "Вперед" між різними станами фільтрів.

* Refresh: При оновленні сторінки (F5) пошуковий запит не скидається.

4) Якщо ви використовуєте метод масиву filter() прямо перед поверненням JSX-розмітки, чи відбувається ця фільтрація при кожному рендері компонента? Як це може вплинути на продуктивність для дуже великих масивів і який інший React-хук ви б використали для оптимізації?
Так, будь-який код (включаючи filter та sort), написаний у тілі функціонального компонента перед return, виконується синхронно при кожному перерендері. Для великих масивів (10 000+ елементів) це призведе до "фрізів" (зависань) інтерфейсу. Для оптимізації необхідно використати хук useMemo, який кешує результат обчислень і перераховує його лише тоді, коли змінюються залежності (сам масив даних або пошуковий запит):
const filteredPosts = useMemo(() => postsData.filter(...), [postsData, searchQuery]);

5) В яких об'єктах стандарту Web API у звичайному (базовому) JavaScript зберігаються параметри після знаку питання ?, і як React Router спрощує роботу з ними?
У нативному Web API для роботи з параметрами запиту використовується об'єкт URLSearchParams (який можна отримати через window.location.search). Робота з ним напряму вимагає написання додаткового коду для оновлення URL без перезавантаження сторінки. React Router надає хук useSearchParams, який повертає масив з двох елементів (подібно до useState): сам об'єкт URLSearchParams для читання даних, та функцію-сеттер, яка автоматично оновлює URL і миттєво ініціює перерендер компонента з новими даними.
