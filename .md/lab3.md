1. Фрагменти коду з логікою роботи пошуку та фільтрації Ця логіка об'єднана у похідній змінній filteredPosts у головному файлі App.jsx

const filteredPosts = postsData.filter(post => {
    const matchesSearch =
      post.content.toLowerCase().includes(searchTerm.toLowerCase()) ||
      post.author.toLowerCase().includes(searchTerm.toLowerCase());
    const matchesCategory = activeCategory === 'All' || post.category
      === activeCategory;
    return matchesSearch && matchesCategory;
  });

2. Пояснення, як реалізовано обробку "порожнього результату" Обробка "порожнього стану" (Empty State) реалізована за допомогою тернарного оператора всередині JSX під час рендерингу блоку feed.

<div className={styles.feed}>
    {filteredPosts.length > 0 ? (
        filteredPosts.map(post => <Post key={post.id} {...post} />)
    ) : (
        <p className={styles.empty}>Нічого не знайдено за вашим
        запитом.</p>
    )}
</div>

Під час рендерингу перевіряється довжина масиву відфільтрованих даних (filteredPosts.length > 0). Якщо в масиві є хоча б один елемент (умова істинна), викликається метод .map(), який генерує список компонентів <Post />. Якщо ж масив порожній (користувач ввів запит, за яким немає жодного збігу), умова повертає хибне значення, і React відрендерить <p> з відповідним текстовим повідомленням про відсутність результатів.


1) Що таке "підняття стану" (lifting state up) і чому ми використовуємо його для SearchBar? 
"Підняття стану" означає переміщення локального стану з дочірнього компонента до їхнього найближчого спільного батьківського компонента. Ми робимо це для SearchBar, оскільки текст пошуку (searchTerm) потрібен саме батьківському компоненту (App), щоб відфільтрувати загальний масив постів перед їх передачею в стрічку новин.


2) Поясніть асинхронну природу оновлення стану в useState. 
Коли викликається функція оновлення стану (наприклад, setSearchTerm), React не перезаписує змінну миттєво. Натомість він ставить це оновлення в чергу і планує перерендеринг компонента. Це дозволяє React оптимізувати продуктивність, групуючи кілька одночасних оновлень станів в один єдиний рендер (batching).


3) Чому для фільтрації ми створюємо нову змінну filteredPosts, а не змінюємо оригінальний масив postsData у стані? 
Оригінальний масив даних є єдиним джерелом правди і має залишатися незмінним (не мутуватися). Якби ми безпосередньо змінили оригінальний масив postsData, ми б назавжди видалили з пам'яті відкинуті пости і не змогли б повернути їх на екран, коли користувач очистить поле пошуку. Саме тому ми завжди створюємо новий масив лише для відображення.


4) У чому перевага використання керованих компонентів над некерованими при реалізації пошуку? 
У керованих компонентах значення інпуту жорстко прив'язане до стану React (через атрибут value). Це дає нам змогу перехоплювати дані миттєво при кожному натисканні клавіші, що є критично важливим для реалізації фільтрації в реальному часі. Крім того, це дозволяє легко очищати поле з коду або застосовувати валідацію.
