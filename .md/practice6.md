* * Відповіді на контрольні запитання
1) В чому полягає критична перевага використання React Developer Tools (Components) порівняно зі звичайним інспектуванням DOM-елементів (Elements)?
Звичайна панель Elements показує лише фінальний результат — HTML-теги та CSS-стилі. Вона не знає нічого про логіку додатку. Натомість вкладка Components у React DevTools показує ієрархію самих React-компонентів, дозволяючи бачити та змінювати їхні поточні props (властивості) та state (стан) у реальному часі, а також переглядати хуки, які в них використовуються. Це дозволяє зрозуміти, чому інтерфейс виглядає саме так, а не просто як він виглядає.

2) Як відловити ситуацію, коли React виконує неоптимальний рендеринг списків через неправильне використання атрибута key?
По-перше, потрібно перевірити стандартну вкладку Console у браузері на наявність попереджень на кшталт Warning: Encountered two children with the same key. По-друге, у налаштуваннях React DevTools (General) можна увімкнути опцію "Highlight updates when components render". Тоді при зміні стану браузер буде підсвічувати зеленими або жовтими рамками ті компоненти, які перемальовуються. Якщо перемальовується весь список, а не лише змінений елемент, проблема у ключах.

3) Поясніть алгоритм дій розробника при встановленні та використанні Breakpoint у вкладці Sources браузера Chrome. Що відбувається з виконанням програми в цей момент?
Алгоритм: відкрити вкладку Sources -> натиснути Ctrl + P (або Cmd + P) -> знайти потрібний файл (наприклад, Feed.jsx) -> натиснути на номер рядка коду, де потрібна зупинка (з'явиться синя позначка).
Коли браузер під час роботи додатку доходить до цього рядка, виконання програми повністю призупиняється (стан "Paused in debugger"). У цей момент розробник може наводити курсор на змінні, щоб побачити їхні значення в пам'яті, та покроково виконувати код далі.

4) Яке ключове слово (резервоване в JavaScript) можна написати безпосередньо у вихідному коді компонента, щоб змусити браузер автоматично зупинити виконання при відкритій панелі розробника?
Це ключове слово debugger;.

5) Що означають дії "Step Over", "Step Into" та "Step Out" у панелі управління дебагером браузера (Sources)?

* Step Over (F10): Виконати поточний рядок коду і перейти до наступного. Якщо на рядку є виклик функції, вона виконається цілком у фоновому режимі (дебагер не буде заходити всередину неї).

* Step Into (F11): Зайти всередину функції, яка викликається на поточному рядку, щоб покроково дослідити її внутрішню логіку.

* Step Out (Shift + F11): Виконати залишок коду в поточній функції і вийти з неї, зупинившись на тому рядку, який її викликав.
