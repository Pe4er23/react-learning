1) Чому при рендерингу масивів у React обов'язково використовувати атрибут key? Що станеться, якщо його не вказати? 
Атрибут key допомагає React ідентифікувати, які саме елементи списку змінилися, були додані або видалені. Це значно оптимізує процес рендерингу (Virtual DOM). Якщо не вказати key, React виведе попередження в консоль, а при зміні порядку елементів або їх видаленні може відбутися некоректне оновлення інтерфейсу (наприклад, стани внутрішніх елементів можуть "перескочити" на сусідні рядки).


2) У чому різниця між методами .map() та .forEach()? Чому .forEach() не підходить для відображення елементів безпосередньо в JSX? 
Метод .map() створює і повертає новий масив з результатами виклику вказаної функції для кожного елемента. Це ідеально підходить для React, бо дозволяє перетворити масив даних на масив JSX-елементів. Метод .forEach() лише виконує функцію для кожного елемента, але завжди повертає undefined. Тому, якщо використати .forEach() всередині JSX, React просто нічого не відрендерить.


3) Чому не рекомендується використовувати індекс масиву (index) як ключ, якщо список може змінюватися (сортуватися, фільтруватися)? 
Якщо використовувати індекс як ключ, то при додаванні, видаленні або сортуванні елементів індекси змістяться. React може "переплутати" компоненти, прив'язавши старий стан інтерфейсу до нових даних. Це призводить до серйозних багів у відображенні. Ключ має бути стабільним та унікальним ідентифікатором самих даних (наприклад, id з бази даних).
